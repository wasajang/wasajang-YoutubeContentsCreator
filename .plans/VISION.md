## 한 줄 요약

**YouTube 크리에이터가 AI를 활용해 시네마틱 영상을 쉽게 제작하고, 자신만의 영상 스타일을 공유·판매할 수 있는 플랫폼.**

---

## 1. 고객 여정 (Customer Journey)

### 3가지 시작점

사용자가 프로젝트를 시작하는 경로는 3가지. 어떤 경로든 선택한 옵션이 저장되어 이후 모든 AI 프롬프트에 반영됨.

```
[A] 대본부터 시작
    → 대본 입력 → 스타일 선택 → Storyboard → Generate → Animate

[B] 영상 스타일(프리셋)부터 시작
    → 프리셋 선택 (Sci-Fi, Horror, Anime 등)
    → 프리셋에 포함된 설정 자동 적용 (프롬프트, 스타일, 추천 캐스트, 비율, AI 모델)
    → 대본 입력 → Storyboard → Generate → Animate

[C] Cast(배우/장소/아이템)부터 시작
    → CastPage에서 AI로 배우 얼굴, 촬영 장소, 소품 이미지 생성
    → 5장 선택 → 대본 입력 → Storyboard → Generate → Animate
```

### 핵심 원칙
- **옵션 영속성:** 어떤 단계에서 선택한 옵션이든 이후 과정에 반영됨
- **자유로운 순서:** 반드시 순서대로 할 필요 없이, 어디서든 시작 가능
- **프리셋 = 시작 가속:** 초보자는 프리셋으로 빠르게, 고급자는 직접 설정

---

## 2. 워크플로우 4단계

```
① Idea        → 대본 작성 + 스타일 선택
② Storyboard  → 카드 선택(덱 구성) + 컷 분할
③ Generate    → 시드 매칭 + AI 이미지 생성 + AI 영상 생성
④ Animate     → 타임라인 편집 + TTS 음성 + Export
```

각 단계에서 사용자의 선택은 store에 저장되어 다음 단계의 AI 프롬프트에 자동 반영.

---

## 3. Cast = AI 에셋 생성 스튜디오

Cast는 단순한 카드 라이브러리가 아닌 **AI 기반 에셋 생성 스튜디오**:

- **배우(캐릭터):** 프롬프트로 얼굴 이미지 생성, Seed로 일관성 유지
- **촬영 장소(배경):** 풍경, 실내, 도시 등 배경 이미지 생성
- **소품(아이템):** 무기, 차량, 도구 등 아이템 이미지 생성

Cast는 **프로젝트에 종속되지 않음** — 여러 프로젝트에서 같은 배우/장소를 재사용.

## 4. AI 모델 선택 시스템

### 4가지 AI 카테고리

AI 모델은 **4개 카테고리**로 나뉘며, 3개는 사용자가 선택 가능하고 1개는 내부 전용:

#### 🔓 사용자 선택 가능 (3개)

| 카테고리 | 용도 | 기본 모델 예시 | 대안 |
|----------|------|--------------|------|
| **대본 AI** | 대본 작성/수정 | GPT-4o Mini | Claude 3 Haiku, Gemini Flash |
| **이미지 AI** | 씬/카드 이미지 생성 | Flux Schnell | SDXL, DALL-E 3 |
| **영상 AI** | 이미지→영상 변환 | Runway Gen-3 | Kling AI, Pika |

#### 🔒 내부 전용 (1개 — 사용자 선택 불가)

| 카테고리 | 용도 | 모델 |
|----------|------|------|
| **프롬프트 생성 AI** | 유저 옵션/프리셋을 조합하여 최적화된 프롬프트 생성 | 내부 최적화 (예: GPT-4o Mini) |

> **프롬프트 생성 AI란?**
> 사용자가 선택한 스타일, 캐스트, 씬 대본, 프리셋 설정값 등을 입력받아
> → 이미지/영상/대본 AI에 전달할 **최적화된 프롬프트를 자동 생성**하는 내부 AI.
> 사용자에게는 보이지 않으며, 플랫폼 내부에서 프롬프트 품질을 제어하는 핵심 엔진.

### 모델 선택 규칙

- **영상 스타일 프리셋이 기본 모델을 지정** (예: "Sci-Fi" → Flux + Runway)
- 사용자가 대본 AI / 이미지 AI / 영상 AI를 자유롭게 변경 가능
- 프롬프트 생성 AI는 Admin에서만 모델 변경 가능 (사용자 UI에 노출되지 않음)
- TTS 음성 (Fish Speech, ElevenLabs 등)은 Animate 단계에서 별도 선택

## 5. 크레딧 시스템 — 2계층 구조

### 핵심 개념

크레딧 비용 = **플랫폼 이용료** + **AI API 비용**

- **플랫폼 이용료:** AntiGravity 워크플로우를 사용하는 기본 비용. BYOK 여부와 무관하게 **항상 차감**.
- **AI API 비용:** 플랫폼이 AI 모델을 대신 호출하는 비용. BYOK 사용 시 **면제** (자기 키로 직접 호출하므로).

### 크레딧 비용표 (예시 — Admin에서 조정 가능)

| 작업 | 플랫폼 이용료 | AI API 비용 | 합계 (기본) | 합계 (BYOK) |
|------|-------------|------------|------------|------------|
| 대본 AI 생성 | 1 | 1 | **2** | **1** |
| 이미지 생성 | 1 | 2 | **3** | **1** |
| 영상 생성 | 2 | 8 | **10** | **2** |
| TTS 음성 | 1 | 1 | **2** | **1** |
| 카드 AI 생성 | 1 | 2 | **3** | **1** |

> 위 숫자는 예시이며, Admin 페이지에서 설정/조정 가능.

### BYOK가 유리한 이유

```
일반 사용자:  10씬 풀 워크플로우 = 약 2 + 30 + 100 + 20 = 152 크레딧
BYOK 사용자:  10씬 풀 워크플로우 = 약 1 + 10 + 20 + 10  =  41 크레딧 (73% 절감)
```

→ BYOK 사용자도 **플랫폼 이용료는 항상 지불** = 플랫폼 매출 보장
→ BYOK 사용자에게 인센티브 (저렴한 비용) → Pro/Enterprise 전환 유도

### 하이브리드 API 모델

| 모드 | 대상 | 크레딧 차감 | 작동 방식 |
|------|------|-----------|----------|
| **플랫폼 API** (기본) | 일반 사용자 | 플랫폼 이용료 + AI API 비용 | 플랫폼이 API 비용 지불 |
| **BYOK** (고급) | 파워유저 | 플랫폼 이용료만 | 유저 자기 API 키로 호출, AI 비용은 유저 부담 |

Settings 페이지에서 서비스별로 전환 가능. 예: 이미지는 BYOK(Replicate 키), 영상은 플랫폼 크레딧.

## 6. Dev Admin

개발자(운영자)만 접근 가능한 관리 페이지:

- **AI 프롬프트 템플릿 편집:** 각 생성 단계별 시스템 프롬프트 관리
- **스타일 프리셋 관리:** 공개(public) / 곧 출시(soon) / 숨김(hidden)
- **AI 모델 설정:** 사용 가능한 모델 목록, 기본 모델 설정
- **크레딧 비용 설정:** 플랫폼 이용료 / AI API 비용 각각 조정 가능
- **시스템 현황:** 사용자 수, 생성량, 크레딧 소비 통계

## 7. 수익 모델

### MVP (Phase 1)
- **무료 플랜:** 100 크레딧/월
- **Pro 플랜:** ₩19,900/월 (1,000 크레딧)
- **Enterprise:** ₩99,000/월 (5,000 크레딧)
- 결제: Stripe + 토스페이먼츠

> 크레딧 단가가 2계층으로 변경되었으므로, 플랜별 크레딧 수량은 실제 비용 시뮬레이션 후 조정 필요.

### 성장기 (Phase 2)
- **UGC 마켓플레이스:** 유저가 자신의 프리셋을 판매
  - 프리셋에 포함되는 것: 대본 생성 프롬프트, 캐릭터 생성 프롬프트, 이미지 생성 프롬프트, 영상 생성 프롬프트(연출), 캐스트 카드 등
  - YouTube 플랫폼 모델: 크리에이터가 영상 스타일을 공유/판매
- **수익 공유:** 다른 유저가 해당 프리셋을 사용하여 크레딧을 소비하면 → 소비된 크레딧의 일부가 프리셋 제작자에게 수익금으로 적립
- **프리셋 작성자 인센티브:** 인기 프리셋 = 지속적 수익

## 8. 기술 방향성

### MVP 기술 스택
- 프론트: React 19 + TypeScript + Vite 7 + Zustand
- 백엔드: Supabase (PostgreSQL + Auth + Storage)
- AI: 멀티 프로바이더 (OpenAI, Replicate, Runway, Fish Speech)
- 배포: Vercel

### 향후 고려
- 서버사이드 API 키 암호화 (Supabase Edge Functions)
- 비동기 작업 큐 (영상 생성 — 웹훅 완료 알림)
- CDN 이미지/영상 캐싱 (Cloudflare R2)
- 모니터링 (Sentry + Vercel Analytics)

## 9. 개발 시 항상 고려할 점

> 아래 원칙을 모든 작업에서 의식하세요.

1. **옵션 영속성:** 사용자 선택은 항상 store에 저장 → 이후 프롬프트에 반영
2. **프리셋 확장성:** 새 프리셋을 데이터만 추가하면 되도록 설계 (코드 수정 최소화)
3. **BYOK 호환:** AI 서비스 호출 시 항상 "BYOK 키 있으면 그걸 쓰고, 없으면 플랫폼 키" 패턴
4. **UGC 준비:** 프리셋/프롬프트 데이터 구조를 나중에 DB 테이블로 이전 가능하게 설계
5. **다중 AI 모델:** 특정 모델에 하드코딩하지 않기 — 모델 ID로 분기하는 구조 유지
