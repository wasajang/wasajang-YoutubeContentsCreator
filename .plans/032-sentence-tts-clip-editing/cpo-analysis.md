## 1. CEO 요구사항 정리 (사용자 여정 기반)

CEO 피드백을 나레이션 워크플로우 단계별로 정리하면 다음과 같습니다.

### 현재 나레이션 워크플로우 (8스텝)

```
Step1: 대본 작성 → Step2: 음성 생성 → Step3: 씬 분할 → Step4: 연출
→ Step5: 이미지 생성 → Step6: 영상화 → Step7: 편집 → Step8: 내보내기
```

### 피드백 매핑

| # | CEO 피드백 | 해당 단계 | 핵심 문제 | 영향 범위 |
|---|-----------|----------|----------|----------|
| 1 | 음성 생성 성공 확인 | Step2 | 없음 (성공) | - |
| 2 | 문장별 개별 음성 파일 생성 | Step2 | 통째 생성 → 문장별 분리 불가 | **아키텍처 변경** |
| 3 | "여기서 나누기" 버그 | Step7 | 엔터키 미작동 | 버그 수정 |
| 4 | Vrew 스타일 편집 | Step7 | 칩/클립 단위 이미지/영상 적용 | **UX 대규모 개편** |
| 5 | 편집 적용 범위 | Step5~7 | 씬 단위 → 클립/칩 단위로 세분화 | **데이터 구조 변경** |

## 2. 사용자 여정으로 본 문제의 본질

### 현재 여정 (문제가 있는 부분)

```
[대본 작성] → [TTS: 전체 대본을 한 덩어리로 음성 생성]
                         ↓
              하나의 오디오 파일 (예: 30초짜리)
                         ↓
              [씬 분할: 시간대로 구간을 잘라서 "씬"으로 분리]
                         ↓
              문제1: 문장 3번만 마음에 안 들어도 전체 재생성 필요
              문제2: 문장별 독립 재생/교체 불가
                         ↓
              [이미지/영상 적용: 씬(=여러 문장 묶음) 단위로만 가능]
                         ↓
              문제3: "이 단어부터 저 단어까지만 다른 이미지" 불가
              문제4: Vrew처럼 세밀한 클립 편집 불가
```

### CEO가 원하는 여정 (Vrew 벤치마크)

```
[대본 작성] → [TTS: 문장별로 개별 음성 파일 생성]
                         ↓
              문장1.mp3, 문장2.mp3, 문장3.mp3, 문장4.mp3
                         ↓
              문장 3번만 재생성 가능 (크레딧 절약 + 시간 절약)
                         ↓
              [클립 구성: 문장들을 묶어서 "클립" 단위로 그룹화]
                         ↓
              [이미지/영상 적용: 클립 단위 또는 칩(단어) 단위로 적용 범위 편집]
                         ↓
              "이 칩부터 저 칩까지" = 자막 한 덩어리 = 이미지/영상 1개 적용
```

### 핵심 인사이트

CEO가 요구하는 것은 단순 기능 추가가 아니라, **편집 단위의 세분화**입니다.

| 현재 | CEO가 원하는 것 | Vrew 기준 |
|------|----------------|----------|
| 1개 오디오 파일 | 문장별 개별 오디오 파일 | 문장별 개별 |
| 씬 단위 이미지 적용 | 클립/칩 단위 이미지 적용 | 클립 단위 |
| 문장 재생성 불가 | 문장별 개별 재생성 | 가능 |
| 나누기 = 씬 분할 | 나누기 = 클립 분할 + 적용 범위 변경 | 클립 분할 |

## 3. Vrew 벤치마크: 우리 서비스에 필요한 핵심 기능

### Vrew의 핵심 개념 (스크린샷 분석)

Vrew는 3계층 구조로 영상을 편집합니다:

```
클립 (Clip)
  └── 칩 (Chip = 단어/어절)
        └── 자막 (= 클립에 속한 칩들의 텍스트)
```

- **클립:** 하나의 이미지/영상이 적용되는 단위. 여러 칩(단어)을 포함.
- **칩:** 개별 단어 또는 어절. 자막의 최소 단위.
- **적용 범위:** 이미지/영상이 보여지는 칩의 범위. 클립 경계에서 변경됨.

### 우리 서비스에 필요한 기능 (우선순위 포함)

| # | 기능 | Vrew 대응 | 우선순위 | 사유 |
|---|------|----------|---------|------|
| A | 문장별 개별 TTS 생성 | 기본 기능 | **필수 (P0)** | 문장별 재생성이 가능해야 편집이 의미있음 |
| B | 문장별 개별 재생/재생성 | 기본 기능 | **필수 (P0)** | A의 자연스러운 연장선 |
| C | 클립 = 이미지/영상 적용 단위 | 클립 개념 | **필수 (P0)** | 현재 "씬"을 "클립"으로 재정의 |
| D | 클립 분할 (나누기) | "새클립으로변경" | **필수 (P0)** | 기존 기능 수정 (버그 포함) |
| E | 클립 합치기 | 클립 병합 | **필수 (P0)** | 기존 기능 유지 |
| F | 적용 범위 변경 | "적용범위변경" | **높음 (P1)** | 칩 단위로 이미지/영상 경계 이동 |
| G | 클립별 이미지/영상 교체 | "교체" | **높음 (P1)** | 클립 선택 후 다른 이미지 적용 |
| H | 칩(단어) 하이라이트 재생 | 단어 하이라이트 | **중간 (P2)** | 재생 중 현재 단어 강조 |
| I | 우클릭 컨텍스트 메뉴 | 스크린샷5 | **낮음 (P3)** | 채우기/애니메이션/순서 등 고급 기능 |
| J | 화자 설정 | 화자 표시 | **낮음 (P3)** | 다중 화자 지원 (Phase 2) |

## 4. 우선순위 판단: MVP 필수 vs 나중에

### MVP 필수 (032에서 반드시 해결)

| 기능 | 사유 | 사용자 가치 |
|------|------|-----------|
| **문장별 개별 TTS** | 전체 재생성은 크레딧 낭비이며, 사용자 통제감을 크게 떨어뜨림 | "이 문장만 다시 만들기" 가능 |
| **클립 = 적용 단위 재정의** | 씬과 클립의 혼용이 사용자 혼란을 야기. Vrew처럼 "클립 = 이미지 1개 단위"로 통일 | 직관적 편집 경험 |
| **나누기 버그 수정** | 기본 기능이 작동하지 않음 | 최소 품질 보장 |
| **클립 분할/합치기 정상화** | 편집의 기본 도구 | 기본 편집 가능 |

### 다음 단계 (033 이후)

| 기능 | 사유 | 시점 |
|------|------|------|
| 적용 범위 드래그 편집 | 복잡한 UI 인터랙션, MVP에서는 분할/합치기로 대체 가능 | 033 |
| 클립별 이미지/영상 교체 (드래그앤드롭) | 분할 후 개별 이미지 생성으로 대체 가능 | 033 |
| 칩 단위 하이라이트 재생 | 이미 단어 타이밍 데이터 있음, UI만 추가하면 됨 | 034 |
| 우클릭 컨텍스트 메뉴 | 편의 기능, 필수 아님 | 035+ |
| 다중 화자 | Phase 2 기능 | 005+ |

## 5. 프로덕트 관점 UX 플로우 제안

### 5-1. 음성 생성 단계 (Step 2) 개선안

**현재:**
```
[전체 대본] → [음성 생성 버튼] → [1개 오디오 파일] → [시간대로 문장 분리 (추정)]
```

**개선안:**
```
[전체 대본] → [문장별 자동 분리 (미리보기)] → [일괄 음성 생성]
                                                    ↓
                                            문장1.mp3 (2.1초)
                                            문장2.mp3 (3.4초)
                                            문장3.mp3 (1.8초)
                                            문장4.mp3 (2.7초)
                                                    ↓
                                    [문장별 재생] [문장별 재생성] [전체 연속 재생]
```

**사용자 가치:**
- "이 문장 발음이 이상해" → 해당 문장만 재생성 (크레딧 1 vs 전체 재생성 크레딧 N)
- 문장별 실제 오디오 길이가 정확히 측정됨 (추정 아닌 실측)
- 문장 순서 변경, 삭제, 추가 후 해당 문장만 재생성

### 5-2. 편집 단계 (Step 7) 개선안 -- Vrew 스타일

**현재:**
```
[좌: 미리보기] + [우: 씬 리스트]
씬 = 여러 문장 묶음, 이미지 1개 적용
```

**개선안: 클립 기반 편집기**
```
[상단: 미리보기 (16:9)] -- 현재 클립의 이미지/영상 + 자막 오버레이
[중단: 클립 스트립] -- 가로 스크롤, 클립별 썸네일 + 칩(단어) 표시
[하단: 타임라인] -- 오디오 파형 + 클립 경계선 + 플레이헤드

클립 = 이미지/영상 1개가 보여지는 구간
칩 = 단어 단위, 자막으로 표시
클립 경계 = 칩 사이에서 드래그로 이동 가능 (적용 범위 변경)
```

**핵심 인터랙션:**

| 동작 | 결과 |
|------|------|
| 클립 클릭 | 미리보기에 해당 클립의 이미지/영상 표시 |
| 칩 사이 "나누기" 클릭 | 클립이 2개로 분할, 각각 별도 이미지 적용 가능 |
| 두 클립 사이 "합치기" 클릭 | 2개 클립이 1개로 병합 |
| 클립의 이미지 클릭 | 이미지 교체/재생성 옵션 |
| 재생 버튼 | 전체 연속 재생, 현재 칩 하이라이트 |

### 5-3. 용어 통일 제안

현재 코드와 UI에서 "씬(scene)"과 "클립(clip)"이 혼재되어 혼란을 줍니다.

| 현재 용어 | 통일 제안 | 의미 |
|----------|----------|------|
| 씬 (Scene) | **클립 (Clip)** | 이미지/영상 1개가 적용되는 편집 단위 |
| 문장 (Sentence) | **문장 (Sentence)** | TTS 음성 1개 단위 |
| 단어/칩 (Word/Chip) | **칩 (Chip)** | 자막의 최소 단위 (단어/어절) |
| 나레이션 클립 | **클립 (Clip)** | 동일 |

> 시네마틱 모드에서는 기존 "씬" 용어를 유지하되, 나레이션 모드에서는 Vrew와 동일하게 "클립" 용어로 통일하는 것을 추천합니다.

## 7. 크레딧 영향 분석

### 현재 (전체 TTS)
- 전체 대본 1회 생성 = 크레딧 2 (플랫폼 1 + AI 1)
- 문장 1개 수정 시 = 전체 재생성 = 크레딧 2

### 개선 후 (문장별 TTS)
- 문장 4개 최초 생성 = 크레딧 2 x 4 = 8 (더 비쌈)
- 문장 1개만 재생성 = 크레딧 2 (1개만 과금)

### 크레딧 정책 제안

| 방식 | 최초 생성 비용 | 부분 재생성 비용 | 추천 |
|------|-------------|----------------|------|
| A. 문장별 개별 과금 | 문장 수 x 2 | 1문장 x 2 | 공정하지만 초기 비용 높음 |
| B. 일괄 생성 할인 | 전체 1회 = 2 | 1문장 x 2 | 사용자 친화적 |
| **C. 하이브리드** | **일괄 = 2, 개별 재생성 = 1** | **1문장 x 1** | **추천: 최초는 저렴, 수정은 합리적** |

> 추천: 방식 C -- 최초 일괄 생성은 기존과 동일한 비용(2 크레딧), 이후 개별 문장 재생성은 1 크레딧. 사용자에게 "일괄이 이득, 수정은 저렴"이라는 명확한 가치 제공.

## 8. 리스크 및 주의사항

| 리스크 | 영향도 | 대응 방안 |
|--------|-------|----------|
| 문장별 TTS API 호출 횟수 증가 → 429 Rate Limit | 높음 | 순차 호출 + 재시도 로직 (이미 구현) |
